<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Module load ↔ external service | `new Client(...)` is constructed at import time; the contract is "no network I/O on construct" — verified in qstash-js@2.10.1 source. Future versions could break this; pin major version. |
| Env var ↔ client config | `QSTASH_TOKEN` flows from `env.ts` → `Client` constructor; never from raw `process.env` outside env.ts |
| Phase 1 ↔ Phase 6 (deferred invocation) | Phase 1 ships the module but does not call any method that makes a network call. Phase 6's planner adds `publishJSON` invocation guarded by signature verification (Receiver class). |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-1-14 | Information Disclosure | QSTASH_TOKEN leaked into client bundle | mitigate | env.ts marks it `server` (T3-Env split prevents bundling). Plus: qstash.ts is server-only by convention (no `"use client"`). ASVS V14. |
| T-1-15 | Tampering | future plan calls `qstash.publishJSON()` without QStash signature verification on the receiver | accept-and-defer | Phase 6's planner is responsible for adding the `Receiver` class (RESEARCH §13 row "QStash signature verification") on every webhook route. Phase 1 only ships the client; deferring to Phase 6 is explicit and correct per CONTEXT D-18. |
| T-1-16 | Spoofing | placeholder/empty token allows construction but blocks runtime calls | accept | The lazy-init contract makes this safe. Phase 1 never calls a method, so an empty token is benign. Phase 6 must validate `env.QSTASH_TOKEN` is set before invoking. |
| T-1-17 | Denial of Service | construction allocates a Client instance even in non-QStash code paths | accept | Single instance per process; memory cost negligible. Module is tree-shakable — code that does not import `@/lib/qstash` does not pay the cost. |
</threat_model>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Build-time vs runtime env | T3-Env's server/client split prevents server vars (DATABASE_URL, NEON_AUTH_COOKIE_SECRET) from being bundled into client JS |
| `.env.example` ↔ secrets | Example file is committed to public repo; values must remain empty placeholders only |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-1-01 | Information Disclosure | server env var leaks to client bundle | mitigate | Use T3-Env's `server`/`client` split; client-only `NEXT_PUBLIC_*` vars; runtimeEnv hand-spread is required by Next.js bundler. ASVS V5/V14. |
| T-1-02 | Tampering | committed `.env.example` accidentally contains a value | mitigate | Augmented file has empty placeholders only; tests/env.example.test.ts can be extended in v1.0 to assert no value follows the equals sign for sensitive keys. Phase 1 acceptance: every line ends with `=` (no inline value) except `QSTASH_URL=https://qstash.upstash.io` (public default) and `NEXT_PUBLIC_APP_URL=http://localhost:3000` (public dev default). |
| T-1-03 | Spoofing | `NEON_AUTH_COOKIE_SECRET` weak (< 32 chars) bypasses HMAC strength | mitigate | `z.string().min(32)` hard-fails import-time when secret is too short. ASVS V6 cryptography. |
| T-1-04 | Tampering | dev contributor sets `SKIP_ENV_VALIDATION=1` and ships a broken build | accept | `skipValidation` is opt-in via env flag; documented as "CI-only". Vercel build does NOT set this flag, so production deploys still validate. |
</threat_model>
